    Task Processing Evolution ‚Äì Threads to Django + Celery

    üß† Background

    Initially, we began integrating a Machine Learning (ML) model into our FastAPI backend to process client requests asynchronously.

    To achieve this, we explored the following Python-native options:

    • threading
    • multiprocessing
    • subprocess

    ‚ùó Problems Encountered

    While this worked for a small number of concurrent requests, the solution didn't scale:

    • High memory and CPU usage during load testing.
    • Threads and processes stayed alive longer than expected.
    • Difficult to manage and monitor.
    • No retry mechanism.
    • No visibility of failed or in-progress tasks.

    üîÅ Introduction of Celery

    To overcome the above problems, we adopted Celery with RabbitMQ (or Redis) as a broker.

    üîß Architecture

    Client ‚Üí FastAPI ‚îÄ‚îÄ‚Üí DB
                   ‚îÇ
                   ‚îî‚îÄ‚îÄ> Try to queue task to Celery
                         ‚îî‚îÄ Fail? ‚Üí Keep in DB
                         ‚îî‚îÄ Success? ‚Üí Mark as 'queued'

    ‚úÖ Pros of Celery

    • Offloads heavy processing from FastAPI.
    • Scalable and distributed.
    • Retry and failure tracking.
    • Integrates well with monitoring tools like Flower.
    • Works with RabbitMQ/Redis as broker, PostgreSQL for result backend.

    ‚ùå Cons of Celery

    • Additional components to maintain (worker, broker, beat).
    • Requires setup for fault tolerance (retry logic, queues, monitoring).
    • Debugging needs external tools/logs.

    üß© Handling Broker Failures

    We designed the logic as follows:

    try:
    my_task.delay(data)
except BrokerUnavailableError:
    save_to_db_as_fallback(data)

    If the Celery broker (RabbitMQ/Redis) is down, we fall back to storing the request in the DB for retry later.

    üîÄ Retry Logic

    Later, a retry worker periodically checks DB for unprocessed tasks and retries pushing them to Celery every 10 minutes using:

    • A simple cron/beat task
    • OR Celery Beat (scheduler)

    ‚úÖ Migration to Django + Celery

    Eventually, we integrated Django + Celery for more robust capabilities and ease of admin.

    Why Django?

    • Built-in ORM and admin interface.
    • Easy to track tasks and statuses using django-celery-results.
    • Easier to manage large-scale tasks and failures.

    üü¢ Pros

    • Admin panel to reprocess/monitor tasks.
    • Built-in tracking via django-celery-results.
    • Easy to configure retries and alerts.
    • DB-backed state for full audit logs.

    üî¥ Cons

    • Heavier than plain FastAPI.
    • More moving parts (Django ORM, models, migrations, etc.).

    üìä Summary Table

    Stage Tools Used Pros Cons
    Initial Threading, Subprocess Simple to implement No retry, memory leaks, unscalable
    Mid FastAPI + Celery Scalable, retry support, async processing Complex setup, broker downtime issues
    Final Django + Celery Admin panel, retry DB, task tracking Heavyweight, Django-specific setup

    ‚úÖ Conclusion

    • For light async tasks: FastAPI + BackgroundTasks is fine.
    • For scalable async processing: Celery is the way to go.
    • For enterprise-level robustness: Django + Celery with result backend and admin tools is ideal.
